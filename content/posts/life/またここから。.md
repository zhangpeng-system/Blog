# ＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝🔼編集必要🔼＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝＝

しばらく止まっていたこのブログを、ようやく再始動することができました。

実は、一度このブログは「壊れて」いました。

３年前の私は、GitHub Actions を使って CI/CD を構築し、Hugo フレームワークをベースにこの静的サイトを公開していました。その時は「動けばいい」という考えが強く、各技術ツールの選定やライブラリのバージョン管理までシステム全体的に深く意識していませんでした。

その後、仕事や新婚生活に追われて長い間ブログから離れてしまい、久しぶりにGitHubから環境を立ち上げようとしたところ、Hugo の仕様変更や依存関係の違いによってプロジェクトのビルドができなくなっていました。

「以前の自分が作ったものなのに、なぜ動かないのか。」

------

## あの頃の自分と、今の自分

３年前の私は、一つひとつの実装や機能に意識が向いていました。
他人のブログと差別化するために、HugoフレームワークとThemeの改造に夢中しました。

しかし、日々の業務を通じて、プロジェクトで設計や保守、運用に携わる中で、少しずつ考え方が変わってきました。

最近では、

- システム全体の構成
- 依存関係の管理
- バージョン固定の重要性
- 将来の保守性
- リスクや課題を先回りして考えること

といった、「長く使い続けられる仕組み」を意識するようになりました。

PMP の学習を通して得たプロジェクトマネジメントの考え方も、この視点に大きく影響しています。

「作ること」よりも、「作ったものを長く動かし続けられること」の方が難しい。そして、それこそが信頼性エンジニアの価値と感じています。

------

## ブログの再構築

今回の再始動では、単純に壊れた部分を修正しただけではありません。
ゼロから全体を見直し、「これから数年間安心して運用できる構成」を目指して再設計しました。

具体的には、

- Hugo のバージョンを固定し、将来も同じ環境で再現できるようにした
- GitHub Actions のワークフローも合わせて整理し、デプロイ手順を明確化
- 使われていなかった Hugo のディレクトリや不要なファイルを整理し、構成をシンプルに
- プロジェクト全体を見直し、保守しやすい形へリファクタリング
- PaperMod の管理方法を Git Submodule から Hugo Modules へ移行

特に最後の変更は、以前から気になっていた課題でした。

Git Submodule は便利な反面、新しい PC で `git clone` しただけではすぐに開発を始められず、初期セットアップにも手間がかかります。

今回 Hugo Modules に移行したことで、依存関係を Go Modules として管理できるようになり、環境構築は以前よりずっとシンプルで再現性の高いものになりました。

------

## AI と一緒にリファクタリング

今回の再構築では、AI の存在も非常に大きな助けになりました。

単にコードを書いてもらうだけではなく、

- 昔の自分が書いたコードを読み解く
- なぜこの構成になっているのかを一緒に分析する
- 不要な処理を洗い出す
- より保守しやすい構成へ改善する

そんな「ペアプログラマー」のような形で活用することができました。

AI のおかげで作業スピードは確かに速くなりました。
しかし、それ以上に感じたのは、「何を改善すべきか」を判断するのは、やはり人間の役割だということです。

AI は強力なツールですが、それをどう使うかは、これまで積み重ねてきた経験や設計の考え方に支えられていると改めて実感しました。

------

## このブログで残していきたいこと

このブログは、技術記事だけを書く場所ではありません。

AWS やクラウド、ソフトウェア開発、プロジェクトマネジメント、生成 AI、そして日本でエンジニアとして働く中で学んだこと。

そうした経験を、自分自身の言葉で記録していきたいと思っています。

数年後に読み返したとき、「あの頃はこんなことを考えていたんだ」と振り返ることができる場所になれば十分です。

そして、もしその記録が誰かの役に立てば、それ以上に嬉しいことはありません。

------

## またここから。

以前のブログを「復元」したのではありません。
今の自分の考え方で、もう一度作り直しました。

エンジニアとして学び続けること。
アウトプットを続けること。
そして、変化を恐れず改善を積み重ねること。

そんな記録を、この場所に少しずつ残していこうと思います。
これから、よろしくお願いします。
